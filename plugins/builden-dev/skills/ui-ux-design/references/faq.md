# 设计常见问题 FAQ

汇总实际开发中常见的设计错误和解决方案。

---

## 组件使用

### Q: 什么时候用 Modal 而不是 Drawer？

**问题**：Modal 和 Drawer 都是弹层，容易混淆。

**解答**：

| 场景                       | 推荐   |
| -------------------------- | ------ |
| 单一操作确认（如删除确认） | Modal  |
| 复杂表单填写               | Drawer |
| 详情查看                   | Drawer |
| 简短提示                   | Toast  |
| 系统级通知                 | Modal  |

**原则**：操作复杂或内容多 → Drawer；简单确认 → Modal

---

### Q: 什么时候不用 Modal？

**反模式**：过度使用 Modal

**问题场景**：

- 列表分页切换用 Modal
- 简单筛选用 Modal
- 每个按钮都弹出确认

**解决方案**：

- 页面内完成简单操作
- 使用 Drawer 承载较复杂内容
- Toast 用于非关键反馈

---

### Q: 表单验证应该实时还是提交后？

**问题**：用户输入时频繁报错，体验差。

**建议**：

- **实时验证**：格式错误（邮箱、手机号）
- **失焦验证**：必填项检查
- **提交验证**：业务逻辑验证

**代码示例**：

```jsx
<Form.Item
  rules={[
    { required: true }]  // 失焦时触发
>
  <Input onBlur={validateField} />
</Form.Item>
```

## 状态设计

### Q: 加载状态到底要不要显示？

**问题**：接口太快，加载状态闪烁；接口太慢，用户等待焦虑。

**解决方案**：

- 延迟显示加载状态（200-300ms）
- 使用骨架屏替代加载动画
- 接口太快时直接显示内容

```jsx
const [showLoading, setShowLoading] = useState(false);

useEffect(() => {
  setShowLoading(true);
  fetchData().finally(() => {
    setTimeout(() => setShowLoading(false), 200);
  });
}, []);
```

---

### Q: 空状态应该展示什么？

**问题**：空列表显示"暂无数据"，用户不知道能做什么。

**好的空状态包含**：

1. 友好插画
2. 清晰文案（如"暂无订单，去逛逛吧"）
3. 操作按钮（如"新建项目"）

**反模式**：

- 只显示"暂无数据"
- 空状态占据整个页面
- 无操作引导

---

### Q: 错误状态如何设计？

**问题**：用户看到错误后不知道怎么办。

**原则**：

1. 清晰错误信息（不是"操作失败"，而是"网络超时，请检查网络"）
2. 提供解决方案
3. 允许重试

**要素**：

- 图标 + 标题 + 描述
- 操作按钮（重试/返回/联系客服）

## 颜色与主题

### Q: 暗色模式颜色怎么调？

**问题**：直接用亮色模式颜色，导致对比度刺眼。

**原则**：

- 背景用深灰（#141414）而非纯黑
- 文本降低透明度（70% 而非 87%）
- 主色提亮（#4096ff 而非 #1677ff）

**检查清单**：

- [ ] 背景不是 #000000
- [ ] 文本对比度 ≥ 4.5:1
- [ ] 边框颜色加深
- [ ] 功能色提亮

---

### Q: 颜色用错了会怎样？

**问题**：功能色使用混乱，用户无法快速理解状态。

**规范**：

- 🟢 绿色 = 成功、完成、通过
- 🟡 黄色 = 警告、注意、待处理
- 🔴 红色 = 错误、失败、危险
- 🔵 蓝色 = 信息、提示、进行中

**反模式**：

- 红色用于"删除"以外的场景
- 绿色用于错误提示
- 同一界面使用超过 4 种颜色

---

## 动效设计

### Q: 动效越长越好？

**问题**：动效太慢让用户觉得卡顿。

**原则**：

| 动效类型        | 时长      |
| --------------- | --------- |
| 微交互（hover） | 150ms     |
| 展开/收起       | 200-300ms |
| 页面切换        | 300-500ms |

**建议**：

- 交互反馈 → 150-200ms
- 状态变化 → 200-300ms
- 页面过渡 → 300-500ms

---

### Q: 所有人都喜欢动效？

**问题**：动效对部分用户造成困扰。

**解决方案**：

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition: none !important;
  }
}
```

## 响应式设计

### Q: 移动端还要适配吗？

**问题**：桌面做好就行，移动端凑合用。

**原则**：

- 移动端触摸目标 ≥ 44px
- 文字可读（不小于 13px）
- 不横向滚动
- 考虑单手操作

---

### Q: 响应式断点怎么选？

**问题**：断点设置混乱，不同组件不一致。

**建议**：

| 断点 | 宽度     | 调整内容       |
| ---- | -------- | -------------- |
| xs   | < 576px  | 手机竖屏，单列 |
| sm   | ≥ 576px  | 手机横屏，2列  |
| md   | ≥ 768px  | 平板，3列      |
| lg   | ≥ 992px  | 小笔记本，4列  |
| xl   | ≥ 1200px | 桌面，6列      |

---

## 可访问性

### Q: 颜色能传达信息吗？

**问题**：只用颜色区分状态，色盲用户无法识别。

**解决方案**：

- 颜色 + 图标
- 颜色 + 文字
- 颜色 + 纹理

**反模式**：

- 只用红色表示错误
- 只用绿色表示成功

---

### Q: 键盘能操作吗？

**问题**：只支持鼠标操作，无法键盘导航。

**检查清单**：

- [ ] Tab 可以切换焦点
- [ ] Enter/Space 触发按钮
- [ ] ESC 关闭弹层
- [ ] 焦点顺序合理

---

## 间距与排版

### Q: 间距要精确吗？

**问题**：间距随意，导致视觉混乱。

**原则**：

- 使用 4px 或 8px 网格
- 同类元素间距一致
- 相关内容靠近

---

### Q: 文字大小怎么选？

**问题**：正文太小（12px），看着费力。

**规范**：

- 正文 ≥ 14px（移动端 ≥ 13px）
- 标题有层级区分
- 使用字重而非字号区分

---

## 总结

### 设计决策checklist

- [ ] 这个组件是必需的吗？
- [ ] 状态都考虑了吗？（加载、空、错误）
- [ ] 动效合理吗？（不过长/不过花哨）
- [ ] 颜色有语义吗？
- [ ] 移动端能用的吗？
- [ ] 键盘能操作吗？
- [ ] 色盲能识别吗？

# 完成前验证

## 概述

没有验证就声称工作完成是不诚实，而不是效率。

**核心原则：** 证据优先于声明，始终如此。

**违反这个规则的字面意思就是违反规则的精神。**

## 铁律

```
没有新鲜验证证据就不能声称完成
```

如果你没有在这个消息中运行验证命令，你就不能声称它通过。

## 门控函数

```
在声称任何状态或表达满意度之前：

1. 识别：什么命令可以证明这个声明？
2. 运行：执行完整命令（全新的、完整的）
3. 阅读：完整输出，检查退出代码，统计失败
4. 验证：输出是否确认声明？
   - 如果否：用证据陈述实际状态
   - 如果是：用证据陈述声明
5. 只有这样：做出声明

跳过任何步骤 = 撒谎，而不是验证
```

## 常见失败

| 声明         | 需要                   | 不充分                      |
| ------------ | ---------------------- | --------------------------- |
| 测试通过     | 测试命令输出：0 个失败 | 上次运行、"应该通过"        |
| Linter 干净  | Linter 输出：0 个错误  | 部分检查、推断              |
| 构建成功     | 构建命令：退出 0       | Linter 通过、日志看起来不错 |
| Bug 修复     | 测试原始症状：通过     | 代码更改、假设修复          |
| 回归测试工作 | 红-绿周期验证          | 测试通过一次                |
| Agent 完成   | VCS diff 显示更改      | Agent 报告"成功"            |
| 需求满足     | 逐行检查清单           | 测试通过                    |

## 红色警示——停止

- 使用"应该"、"可能"、"似乎"
- 在验证之前表达满意度（"太棒了！"、"完美！"、"完成！"等）
- 即将提交/推送/PR 而不验证
- 信任 agent 成功报告
- 依赖部分验证
- 认为"就这一次"
- 疲倦并想要完成工作
- **任何暗示成功但没有运行验证的措辞**

## 防止合理化

| 借口                       | 现实             |
| -------------------------- | ---------------- |
| "现在应该工作了"           | 运行验证         |
| "我有信心"                 | 信心 ≠ 证据      |
| "就这一次"                 | 没有例外         |
| "Linter 通过了"            | Linter ≠ 编译器  |
| "Agent 说成功"             | 独立验证         |
| "我累了"                   | 疲惫 ≠ 借口      |
| "部分检查就够了"           | 部分证明不了什么 |
| "不同的措辞所以规则不适用" | 精神重于字面     |

## 关键模式

**测试：**

```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过了" / "看起来正确"
```

**回归测试（TDD 红-绿）：**

```
✅ 写 → 运行（通过）→ 恢复修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有红-绿验证）
```

**构建：**

```
✅ [运行构建] [看到：退出 0] "构建通过"
❌ "Linter 通过了"（linter 不检查编译）
```

**需求：**

```
✅ 重读计划 → 创建检查清单 → 验证每个 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**Agent 委托：**

```
✅ Agent 报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任 agent 报告
```

## 为什么这重要

来自 24 个失败记忆：

- 你的合作伙伴说"我不相信你"——信任被破坏
- 未定义的函数被发货——会崩溃
- 缺失的需求被发货——功能不完整
- 虚假完成浪费时间 → 重定向 → 返工
- 违反："诚实是核心价值。如果你撒谎，你会被替换。"

## 何时应用

**始终在之前：**

- 任何成功/完成的声明变体
- 任何表达满意度的陈述
- 任何关于工作状态的正向陈述
- 提交、PR 创建、任务完成
- 转向下一个任务
- 委托给 agents

**规则适用于：**

- 精确短语
- 改写和同义词
- 成功的暗示
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后声明结果。

这是不可协商的。

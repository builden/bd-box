# 代码质量标准定义

> 参考：王争《设计模式之美》

## 核心评价标准

最常用、最重要的三个评价标准：**可维护性 > 可读性 > 可扩展性**

```
┌─────────────────────────────────────────────────────────┐
│                    代码质量金字塔                        │
├─────────────────────────────────────────────────────────┤
│                    可维护性 (最高)                       │
│        ┌─────────────────────────────────┐             │
│        │         可读性                   │             │
│        │    ┌─────────────────────┐       │             │
│        │    │    可扩展性          │       │             │
│        │    │  ┌───────────────┐  │       │             │
│        │    │  │  简洁性       │  │       │             │
│        │    │  │可复用性|可测试性│  │       │             │
│        │    │  └───────────────┘  │       │             │
│        └─────────────────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

---

## 1. 可维护性 (Maintainability)

### 定义

代码在**不破坏原有设计**、**不引入新 bug**的前提下，能够**快速修改或添加代码**。

### 好 vs 不好

| 好的代码                     | 不好的代码                      |
| ---------------------------- | ------------------------------- |
| 修改功能只需改动 1-2 个模块  | 修改一个功能需要改 10+ 个文件   |
| Bug 定位快速，修复后不易复发 | 修 Bug 引入新 Bug，陷入恶性循环 |
| 新人上手快，1 周能独立改代码 | 新人上手 1 个月仍不敢碰代码     |
| 代码结构清晰，模块职责明确   | 代码耦合严重，牵一发而动全身    |

### 影响要素

- 可读性
- 简洁性
- 可扩展性
- 模块化程度（高内聚低耦合）
- 分层是否清晰

---

## 2. 可读性 (Readability)

### 定义

**代码是写给人看的**，不是写给机器看的。代码被阅读的次数远超过编写次数，读不懂就无法安全地修改。

### 好 vs 不好

| 好的代码                           | 不好的代码                         |
| ---------------------------------- | ---------------------------------- |
| 命名自解释：`calculateUserAge()`   | 命名模糊：`getData()`、`process()` |
| 函数长度 < 30 行                   | 函数 200+ 行，一个屏幕装不下       |
| 注释说明**为什么**，而非**是什么** | 注释：`// 循环`（这不废话吗）      |
| 嵌套不超过 3 层                    | 嵌套 6+ 层，箭头代码               |
| 单一职责，一个函数只做一件事       | 一个函数做 5 件事                  |

### 评判方法

- **Code Review**：同事能否轻松读懂你的代码
- **新人测试**：新人能否快速上手
- **自问自答**：一个月后自己还能看懂吗

---

## 3. 可扩展性 (Extensibility)

### 定义

**应对未来需求变化的能力**——在不修改或少量修改原有代码的情况下，添加新功能。

### 好 vs 不好

| 好的代码                           | 不好的代码                       |
| ---------------------------------- | -------------------------------- |
| 预留扩展点：接口、抽象类、插件机制 | 硬编码判断，添加功能就改核心代码 |
| 依赖注入：不依赖具体实现           | 强耦合具体类，新功能无法替换     |
| 遵循 OCP：对扩展开放，对修改关闭   | 新需求来就改老代码               |
| 职责单一：改动点集中               | 一个类管太多，改动点分散         |

### 实现手段

- 基于接口而非实现编程
- 依赖注入
- 装饰者模式、策略模式
- 预留抽象层

---

## 4. 简洁性 (Simplicity)

### 核心原则：KISS - Keep It Simple, Stupid

### 好 vs 不好

| 好的代码       | 不好的代码            |
| -------------- | --------------------- |
| 200 行解决战斗 | 2000 行 "过度设计"    |
| 直截了当的逻辑 | 绕来绕去的"技巧"      |
| 朴素直白的方式 | 炫技式的奇技淫巧      |
| 最小依赖       | 引入 10 个库做 1 件事 |

### 反例警示

```typescript
// ❌ 过度工程
class UserServiceImpl implements IUserService, IUserRepository,
  ICacheManager, IEventPublisher, ILoggerAware { ... }

// ✅ 简单直接
function getUser(id: string): User | null { ... }
```

---

## 5. 可复用性 (Reusability)

### 定义

**减少重复代码**，复用已有的模块/函数/类。

### 好 vs 不好

| 好的代码               | 不好的代码             |
| ---------------------- | ---------------------- |
| 通用逻辑抽象为工具函数 | 复制粘贴 10 份相似代码 |
| 组件库复用             | 每个项目重写轮子       |
| 继承/组合复用          | 重复造轮子             |
| 库级别抽取             | 业务代码直接写死       |

### 实现手段

- 函数抽取
- 组件抽象
- 继承/组合
- 库/包发布

---

## 6. 可测试性 (Testability)

### 定义

代码是否容易被**单元测试**覆盖和验证。

### 好 vs 不好

| 好的代码                 | 不好的代码                   |
| ------------------------ | ---------------------------- |
| 纯函数：无副作用，易测试 | 依赖全局状态、文件系统、网络 |
| 依赖注入：可 mock        | 硬依赖具体实现               |
| 单一职责：测试目标明确   | 一个函数测 100 种情况        |
| 单元测试覆盖 80%+        | 没测试 or 只有集成测试       |

### 实践原则

- 依赖注入
- 避免 static 方法、全局状态
- 函数保持纯度
- 提前考虑可测试性

---

## 7. 灵活性 (Flexibility)

### 定义

代码的**适应性**——易扩展、易复用、易组合。

### 与其他标准的关系

- 可扩展性 → 灵活性
- 可复用性 → 灵活性
- 接口抽象 → 灵活性

### 好 vs 不好

| 好的代码           | 不好的代码     |
| ------------------ | -------------- |
| 插件化架构         | 写死的配置     |
| 策略模式可切换算法 | 硬编码 if/else |
| 中间件机制         | 耦合的业务逻辑 |
| 配置驱动           | 代码写死       |

---

## 8. 高性能 (Performance)

### 定义

代码执行效率高，资源占用低。

### 注意

- **不要过早优化**：先跑通，再优化
- **测量驱动**：用 profiler 找瓶颈
- **权衡取舍**：可读性 > 小幅性能提升

### 好 vs 不好

| 场景 | 好的做法          | 不好的做法     |
| ---- | ----------------- | -------------- |
| 循环 | 提前 break/return | 无条件遍历全部 |
| 缓存 | 避免重复计算      | 每次重新算     |
| I/O  | 异步/批量         | 同步串行       |
| 内存 | 及时释放          | 内存泄漏       |

---

## 9. 安全性 (Security)

### 好 vs 不好

| 好的代码     | 不好的代码     |
| ------------ | -------------- |
| 输入验证     | 不校验用户输入 |
| 参数化查询   | 字符串拼接 SQL |
| 环境变量配置 | 硬编码密钥     |
| 权限校验     | 无权限控制     |

---

## 10. 可观察性 (Observability)

### 定义

系统运行状态可观测——日志、监控、追踪。

### 实践

```typescript
// ✅ 结构化日志
logger.info("order_created", { orderId: "123", amount: 100 });

// ❌ 字符串拼接
console.log("订单创建成功");
```

## 标准之间的关系

```
可读性 ──┬──> 可维护性
         │
简洁性 ──┤
         ├──> 可扩展性
可复用性 ┤
         │
可测试性 ┴──> 灵活性
```

**核心洞察**：

- 可维护性是**最高目标**——其他标准都是达成它的手段
- 可读性是**基础**——读不懂就无法维护
- 简洁性是**方法**——简单代码更容易维护和扩展

---

## 评判代码质量的原则

1. **多维度综合评价**：不能只看单一维度
2. **主观 + 客观**：经验判断 + 具体指标
3. **团队共识**：Code Review 形成统一标准
4. **持续迭代**：没有完美代码，只有不断改进

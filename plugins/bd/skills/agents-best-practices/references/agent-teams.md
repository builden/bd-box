# Agent Teams

Agent Teams 是实验性功能，用于协调多个 Claude Code 实例协同工作。

## 启用 Agent Teams

Agent Teams 默认禁用。在 `settings.json` 中启用：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

## Agent Teams vs Subagents

| 特性           | Subagents                        | Agent Teams              |
| -------------- | -------------------------------- | ------------------------ |
| **上下文**     | 独立上下文窗口，结果返回给调用者 | 独立上下文窗口，完全独立 |
| **通信**       | 只能报告结果给主 Agent           | 队友之间直接消息         |
| **协调**       | 主 Agent 管理所有工作            | 共享任务列表，自主协调   |
| **适用场景**   | 只需要结果的专注任务             | 需要讨论和协作的复杂工作 |
| **Token 成本** | 较低                             | 较高                     |

## 何时使用 Agent Teams

Agent Teams 最有效的场景：

- **研究和审查**：多个队友同时调查问题的不同方面
- **新模块或功能**：每个队友独立负责不同部分
- **调试竞争假设**：队友并行测试不同理论
- **跨层协调**：前端、后端和测试的变化

## 创建 Agent Team

启用后，用自然语言描述任务和团队结构：

```
我正在设计一个帮助开发者跟踪 TODO 评论的 CLI 工具。
创建一个 Agent Team 从不同角度探索：一个负责 UX，
一个负责技术架构，一个扮演反对派。
```

## 控制 Agent Team

### 显示模式

- **In-process**：所有队友在主终端内运行
- **Split panes**：每个队友有独立窗格

在 `settings.json` 中配置：

```json
{
  "teammateMode": "in-process" // 或 "tmux"
}
```

### 任务分配

共享任务列表协调团队工作：

- **队长分配**：告诉队长把任务交给哪个队友
- **自领**：队友完成后自行领取下一个未分配任务

### 与队友直接对话

- **In-process 模式**：按 Shift+Down 循环切换队友
- **Split-pane 模式**：点击队友窗格直接交互

### 关闭队友

```
让 researcher 队友关闭
```

### 清理团队

```
清理团队
```

## 最佳实践

### 给队友足够的上下文

队友自动加载项目上下文，但不继承队长的对话历史。在生成提示中包含任务细节：

```
生成一个安全审查员队友，提示："审查 src/auth/ 中的认证模块
的安全漏洞。关注令牌处理、会话管理和输入验证。"
```

### 选择合适的团队规模

- Token 成本线性增长
- 协调开销增加
- 3-5 个队友适合大多数工作流

### 任务大小适当

- **太小**：协调开销超过收益
- **太大**：队友长时间工作无检查，风险增加
- **合适**：自包含的单元，产生明确交付物

### 等待队友完成

有时队长自己开始实现任务而不是等待队友。如果注意到这种情况：

```
等待你的队友完成任务后再继续
```

## 限制

Agent Teams 是实验性功能，当前限制：

- In-process 队友不支持会话恢复
- 任务状态可能滞后
- 关闭可能较慢
- 每个会话只能有一个团队
- 队友不能生成自己的团队
- 队长固定，无法转移
- Split panes 需要 tmux 或 iTerm2

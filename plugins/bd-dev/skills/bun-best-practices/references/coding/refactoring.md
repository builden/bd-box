# 重构规范

本文档定义了大型重构项目（如 JavaScript → TypeScript 迁移）的标准和流程。

## 概述

大型重构是指涉及多个文件、可能影响系统行为的代码转换。与日常小改bug不同，大型重构需要：

1. 明确的计划
2. 进度追踪
3. 多层次验证
4. 安全清理

---

## 重构计划文件

在项目根目录创建 `REFACTORING_PLAN.md`：

```markdown
# 重构计划文件

## 重构目标

- 技术目标：xxx
- 质量目标：xxx

## 重构原则

1. 进度管理：每个阶段必须验证通过才能进入下一阶段
2. 测试优先：先写测试，再写实现
3. 清理规则：所有测试通过后才能清理旧文件

## 重构阶段

- 阶段1: 静态分析测试 → 验收标准
- 阶段2: 单元测试 → 验收标准
- 阶段3: 服务启动 → 验收标准
  ...

## 当前状态

- 已完成：阶段1、阶段2
- 进行中：阶段3
- 待处理：阶段4-7

## 已知问题

- 问题1: 描述...
```

---

## 进度追踪文件

在项目根目录创建 `REFACTORING_PROGRESS.md`。

### 更新规则

**每完成一个小阶段就更新**，而不是：

- 遇到问题才更新
- 所有工作做完才更新

### 更新时机

```
✓ 完成一个任务
✓ 发现一个问题
✓ 进入下一阶段
✓ 遇到阻塞
```

---

## 重构阶段

必须按顺序执行，每个阶段验证通过后才能进入下一阶段：

### 阶段 1: 静态分析测试

- **目标**: 验证 JS 文件结构完整性
- **验收**: 所有静态分析测试通过
- **耗时**: 快速（秒级）

### 阶段 2: 单元测试

- **目标**: 验证每个模块的功能
- **验收**: 所有单元测试通过
- **耗时**: 快速（分钟级）

### 阶段 3: 服务启动测试

- **目标**: 服务能正常启动
- **验收**: `bun run server` 无报错，端口监听
- **耗时**: 中速

### 阶段 4: API 端点测试

- **目标**: 核心 API 可用
- **验收**:
  - GET /api/auth/status → 200
  - GET /api/projects → 200
  - POST /api/auth/login → 200
- **耗时**: 中速

### 阶段 5: 集成测试

- **目标**: 关键功能正常工作
- **验收**:
  - 终端能连接
  - Claude 会话能创建
  - WebSocket 通信正常
- **耗时**: 较慢

### 阶段 6: E2E 测试

- **目标**: 完整用户流程
- **验收**: 端到端测试通过
- **耗时**: 慢

### 阶段 7: 清理旧文件

- **目标**: 删除旧的源文件
- **验收**: 服务仍然正常工作
- **注意**: 必须等所有阶段通过后才能清理

---

## 测试分层与优先级

### P0 - 阻塞发布（必须通过）

```
- 服务能启动
- 核心 API 可用（auth, projects）
- 关键业务流程不断裂
```

### P1 - 影响体验（重要）

```
- 所有单元测试通过
- 静态分析测试通过
- WebSocket 连接正常
```

### P2 - 锦上添花

```
- 边界条件处理
- 错误处理
- 性能测试
```

### 测试顺序

1. 静态分析测试（验证结构）→ 快速
2. 单元测试（验证函数）→ 快速
3. 服务启动（验证集成）→ 中速
4. API 测试（验证端点）→ 中速
5. E2E 测试（验证流程）→ 慢速

---

## 质量门禁

发布前必须全部通过：

```
□ 门禁1: bun test → 100% pass
□ 门禁2: bun run server → 无报错，端口监听
□ 门禁3: API 冒烟测试 → 全部 200
□ 门禁4: 关键功能验证 → 手动/自动化通过
□ 门禁5: bun lint && bun typecheck → 无错误
```

---

## 清理规则

### 永远不要这样做

- 重构进行中就删除旧文件
- 测试没通过就删除旧文件
- 没有备份就删除旧文件

### 正确的清理流程

1. **确认阶段 6 通过后**再考虑清理
2. **先移动备份**：`mv xxx.js xxx.js.bak`（不立即删除）
3. **观察 24 小时**后再决定是否删除备份
4. **保留一周**后再清理备份文件

---

## TDD 流程

每个文件重构必须遵循：

```
1. 分析原 JS 文件功能
   ↓
2. 编写测试用例（静态分析/行为验证）
   ↓
3. 运行测试（应该失败）
   ↓
4. 复制 JS → TS
   ↓
5. 运行测试（应该通过）
   ↓
6. 验证功能
```

## 调试方法

重构过程中遇到问题时，遵循调试优先级：

```
1. 日志 → 2. 测试 → 3. 源码
```

**不是直接分析源码**，而是：

- 先用测试用例复现问题
- 测试即调试：把问题转化为测试用例，既能定位又能防止回归
- 详细规范见 [testing.md](./testing.md)

## 常见问题与解决方案

### 问题 1: Placeholder 文件

**现象**: 项目中存在空的 .ts placeholder 文件，运行时找不到模块

**原因**:

- 之前会话创建的空文件
- 复制时没有检查内容

**解决**:

- 复制前检查文件内容：`head -5 xxx.ts`
- 确保复制的是完整代码

### 问题 2: 路由导入路径

**现象**: 终端卡住，Claude 会话无法创建

**原因**:

- routes/\*.ts 仍在导入 .js 文件
- 模块路径指向错误

**解决**:

- 检查所有导入路径：`grep -r "from.*\.js'" server/routes/`
- 更新为 .ts 后缀

### 问题 3: 进度失控

**现象**: 不知道做到哪了，离目标还有多远

**原因**:

- 没有明确的里程碑
- 没有验收标准

**解决**:

- 创建 REFACTORING_PLAN.md
- 创建 REFACTORING_PROGRESS.md
- 每个阶段验证通过才能进入下一阶段

### 问题 4: 环境差异导致功能异常

**现象**: 重构后终端功能不正常，但测试通过

**原因**:

- 测试只覆盖了简化场景（plain shell），没有覆盖真实场景（Claude CLI）
- 测试没有验证后台进程是否真正运行
- 开发环境有特殊配置（代理、PATH、NODE_PATH 等）

**解决**:

- 测试必须覆盖真实用户场景
- 添加进程状态验证
- 在调试时检查环境变量：`env | grep -E "PROXY|PATH|NODE"`

```typescript
// ✅ 测试真实场景
it('should work with real Claude CLI', () => {
  ws.send({
    type: 'init',
    provider: 'claude', // 不是 plain shell
    isPlainShell: false,
  });
  // 验证 PTY 进程启动
  expect(await isProcessRunning('spawn-helper')).toBe(true);
});
```

### 问题 5: 重构引入微妙的环境问题

**现象**: 环境变量在新场景下不可用，导致功能失败

**原因**:

- 看似无关的环境变量实际被依赖
- 清理环境变量时过度激进

**解决**:

- 保留必要的环境变量（PATH、NODE_PATH 等）
- 只清理明确有问题的变量
- 使用排除而非全部清理的方式

```typescript
// ❌ 过度清理
const cleanEnv = {};
for (const key of Object.keys(process.env)) {
  cleanEnv[key] = process.env[key]; // 保留所有
}

// ✅ 精确清理
const cleanEnv = { ...process.env };
delete cleanEnv.CLAUDECODE;
delete cleanEnv.HTTP_PROXY;
```

## 检查清单

重构前：

```
□ 列出所有待重构文件
□ 制定重构顺序（依赖关系）
□ 确定验收标准
```

重构中：

```
□ 测试先行（每个文件）
□ 验证后进入下一步
□ 遇到问题立即报告
```

重构后：

```
□ 单元测试 100% 通过
□ 服务能正常启动
□ API 端点测试通过
□ 关键功能手动验证
□ 代码 lint + typecheck 通过
□ 清理旧文件（观察期后）
```

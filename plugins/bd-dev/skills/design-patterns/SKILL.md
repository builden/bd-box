# 设计模式 Skill

本 skill 帮助开发者在写代码时选择合适的模式，理解为什么好，什么场景适用，什么时候不该用。

## 触发场景

- "用什么模式好"
- "这段代码用什么设计模式"
- "代码太耦合了"
- "需要扩展"
- 提及具体模式名（策略、观察者、单例等）

## 设计原则

### SOLID 原则

| 原则     | 一句话                 | 核心价值       |
| -------- | ---------------------- | -------------- |
| 单一职责 | 一个类只做一件事       | 可维护、可测试 |
| 开闭     | 对扩展开放，对修改封闭 | 可扩展、可复用 |
| 里氏替换 | 子类可以替换父类       | 可靠性         |
| 接口隔离 | 尽量少依赖小接口       | 解耦           |
| 依赖倒置 | 依赖抽象，不依赖具体   | 可替换         |

### 辅助原则

- **迪米特法则**：只和直接朋友交流
- **组合优于继承**：多用组合，少用继承
- **命令查询分离**：命令不返回数据，查询不改变状态
- **好莱坞原则**：别调用我们，我们会调用你

## 快速选择

| 你的需求                 | 推荐模式             |
| ------------------------ | -------------------- |
| 需要一个全局唯一实例     | 单例                 |
| 创建对象逻辑复杂         | 工厂/抽象工厂/建造者 |
| 对象的创建成本高，想复用 | 原型/享元            |
| 需要统一接口             | 适配器/外观          |
| 动态添加功能             | 装饰器               |
| 处理树形结构             | 组合                 |
| 远程对象本地用           | 代理                 |
| 算法可替换               | 策略                 |
| 状态变化自动响应         | 观察者/状态          |
| 操作可撤销               | 命令/备忘录          |
| 遍历集合                 | 迭代器               |
| 多个对象需要协调         | 中介者               |
| 行为随状态变化           | 状态                 |
| 处理多步骤流程           | 模板方法/职责链      |
